Accman Refresh : Thoughts On Approaching A Refresh Of Accman

Created by Unknown User (uis612), last modified on Aug 07, 2014

Introduction

This is a fairly stream-of-consciousness page that identifies some of the high-level steps required to re-imagine Accman as a stable system that can be easily maintained.

Identify Inputs And Outputs

Rather than look at what Accman currently does and work backwards, it is suggested that the inputs to and outputs from the existing system are examined first. Then, the appropriate transformations that occur between them can be examined in a subsequent step.

Inputs

Accman reads data from the following systems, which are the sources of users that require access to university facilities.

SystemUser TypeDescription
BannerStudentsStudent Information System (S.I.S.)
S.A.P.StaffPayroll and human resources system
AssociatesAssociates

Maintains individuals who are regarded as neither students nor staff,

(i.e. contractors, conference attendees, etc.)

Additional Questions

  • Is the above list complete? Do other systems act as a source of account users?
  • What about functional accounts?
  • What about group accounts?
  • Should 'generic' accounts (i.e. those that do not correspond to a single individual) be handled by Accman, or should they remain in their source system, i.e. Active Directory?

Accman writes data to the following systems, which provide facilities to the account users sourced from the above systems.

Additional Questions

  • The above list is far from exhaustive. What other systems does Accman write to?

Identify Data Points

This stage comprises the identification of both common and unique data elements from each of the input systems. This step could also usefully include highlighting elements (.e.g. 'department') that are represented differently on the input systems, with a view to creating a mapping set. Continuing the previous example, the 'departments' held on Banner may differ from the 'departments' held in S.A.P. It is likely that the level of granularity between the input systems could also differ, i.e. 'departments' and 'colleges' in Banner may not directly map to 'departments' and 'divisions' in S.A.P.

Where common elements are identified, analysis is required of data types and sizes used by all systems. Thus, a common denominator can be reached, i.e. a 60-character surname on Banner and a 75-character last name on S.A.P. could be mapped to a 100-character family name in Accman.

Identify Data Transformations

Immediately following the identification of the data points to be managed, consideration is required of the transformations of this data that will occur. This will entail reviewing all the existing Accman code and abstracting out each logical process. Example process may include adding a new student, graduating a student, employing a new staff member, retiring a staff member, and so on.
