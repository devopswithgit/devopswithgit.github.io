Introduction

Please also refer to the official project documentation.

Accman is to be modified in the following ways:

  1. New Fields. Students only, the "Personal Email" and "Personal Contact Phone Number" fields need to be retrieved from Banner during the Banner Import, and stored in Accman
  2. Update Frequency. For Staff, Students and Associates, Accman should update ALMA whenever someone's 'person' data changes.

New Fields

The "Personal Email" and "Personal Contact Phone Number" information is stored in Banner. The exact field names have yet to be confirmed. They will be retrieved from Banner during Accman's Banner Import process, and stored in Accman's Person table alongside the other personal information currently held. 

Banner Query

The query currently used to retrieve the Banner data is given below. This will be modified to retrieve the new fields when the table and column names are known.

The updated SQL includes a necessary join on a view called SALFORD.V_SAL_LATEST_INTL_ACCESS_FULL that (at least in testing) increases the query execution time to around 6-8 minutes. This may require an adjustment in the timings of the Banner Import process to give it time to complete before subsequent stages begin.

Banner Import Process (Overview)

The Banner Import process runs every 2 hours between 9:30am and 7:30pm Monday-Thursday and between 9:30 and 5:30 on Fridays. For each student it finds, it compares the data already stored in Accman with what has just been retrieved. If a change is detected (for example a change in Surname as a result of getting married), the change is recorded and an "Update Janus" transaction (type 97) is generated. The process is shown in the diagram below.

Changes Required

In order to support the new fields, the following key changes need to be made:

  1. Modify Banner SQL in BannerDAO

  2. Add the new columns (PersonalPhone, PersonalEmail) to the BANNER_IMPORT table in Accman

    Add columns to BANNER_IMPORT table
    alter table banner_import add GOREMAL_EMAIL_ADDRESS varchar(128);
alter table banner_import add MOBILE_NUMBER varchar(22);

  3. Add the new columns (PersonalPhone, PersonalEmail) to the PERSON table in Accman

    Add columns to PERSON table
    alter table person add personal_email varchar(128);
alter table person add mobile_number varchar(22);
  4. Add the new fields to the Person class (C# code)
  5. Ensure that all other code involved in the process shown in the above diagram takes the new fields into account, including BannerDAO, BannerImportDao, PersonDAO, AccountDAO and StudentActions. 
  6. The new fields need adding to the ALMA Rest Interface. This needs to be added to the ContactInfo class (there are already fields for this (Email & Phone). These fields are List objects and will be structured in the following way

Update Frequency

Currently, ALMA updates only take place when a user's ID card details change. For example, if the user loses their card and a new one is issued, Janus will signal to Accman that this has happened, and Accman generates an "Update ALMA" transaction (type 127) which is subsequently processed. Pushing updated details to ALMA this way is not frequent enough because someone's personal details could change but their card details could remain the same for years, so their changed information would never be pushed into ALMA.

To alleviate this problem, there is a monthly batch job to run an ALMA update for every user in the system. It runs on the 15th of each month and takes several hours to complete. This is not an ideal solution though, because if someone's surname changes on the 16th (for example), then this change will still have to wait almost an entire month before it is pushed into ALMA. 

This change involves pushing updates through to ALMA much more frequently, almost as soon as the changes occur. For staff and students, this is achieved by modifying the SAP Import and Banner Import processes respectively. For Associates, it requires changing the Accman web application.

Staff

The SAP Import runs twice a day. If it detects that someone's details have changed it creates an Update Janus transaction (type 97). This is the logical place to update ALMA too. The code to change is in method SapPersonImport.UpdatePerson. It currently checks the following fields for changes:

Initials, ForeName, RealName (Surname), DepartmentCode, PersonState, Title, PreferredName, StartDate, EndDate, BirthDate, SecretKey, MultiPost

If any of these fields coming from SAP differ from their values held in Accman at the time of the import, the Update Janus Transaction is executed by the following code:

Associates

 

Updates to associate accounts are not automated. However their details can be modified via the 'Manage Associates' screen in the Accman web application (see below). A member of ITS ServiceDesk is likely to make these modifications when a user calls with a request to change their details. An Update ALMA transaction should therefore be created when the user presses "Save".

The change should be made to the AssociateController.CreateOrEdit method and should include:

  • Only check the fields for changes if an existing user is being edited, not a new user being created.
  • Check for field changes for both standard associates and Cohort Group associates.
  • If a change is detected, generate an Update ALMA transaction using the standard TransactionDAO methods
