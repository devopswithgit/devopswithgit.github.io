Details

  • Script to generate transaction 1 and other helper scripts to check available accounts. Found in:

    Q:\Account Manager\support\useradmin\pregeneration of accounts\pregen.pl

    Q:\Account Manager\support\useradmin\pregeneration of accounts\available.pl

We should find a way of replacing the perl or automating the process.

History

Pregen.pl was originally written to initiate Pregen transaction(#1). Transaction#1(Pregen) generates a specified number of account in AccMan with a specified 3-letter username prefix. We have already translated most perl scripts in .Net (AccMan.NET). However there are still few handy perl scripts remains to convert and they are used by support team. One of them is pregen.pl, don't be confuse with pregen transaction#1.

Problem

Historically support team used pregen to pre populate blank account in AccMan. which is then used by banner to assign PIDM and eventually person records will be created and assigned to the account. Idea of pre-populate blank account works fine. However first steps of this process is manual and need human intervention and keeping entire history of all 3-letter username prfix in Excel file (which is not ideal and safe). There is also limit set to username letter to only 6. This restriction sometime runout of 3-letter department code and forced to change it. For example UIS was used to be prefix for all users belong to ITS. As limit of 999 ran out and prefix changed to UJS.

Another problem is for every department in AccMan, support team tend to create 50 odd blank account. Which works for some department with more users. However some department hardly have any user and this leave many blank account remain in AccMan. As OPS continues to improve security and infrastructure, these pre populated blank user account sometime cause an issue. For example recently OPS changes password policy which failed as AccMan used to create account with 6 digit password does not match the policy.

Solutions

Option-1

Remove pregen process completely and creates user account on the fly.

Option-2

Remove manual pregen process and make it automatic and leave the rest of the steps as it is.

On top of these two options, we should also remove the restriction of 999 suffix in username. I have checked with OPS and they have confirmed that there is no such restriction with username must be 6 characters long. Only reason we opted to have 6 character username was to easy to remember. As we are growing rapidly, its time to remove this restriction. One option could be to change 999 to 9999, this will give us enough account for next couple of years.
