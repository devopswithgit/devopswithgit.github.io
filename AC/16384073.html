Introduction

As well as the traditional Perl processes that run on the Accman production servers, we now have lots of .NET processes that also run on the same machine.

These perform various tasks such as:

  • Creating a user account in ActiveDirectory
  • Changing Novell / AD passwords
  • Importing data into Janus
  • Sending emails
  • Setting up associate accounts

They work in very much the same way as the Perl processes, but writing them in .NET allows us to make use of many features that .NET offers. A full list with descriptions is provided at the end of this document.

How do they work?

The processes are all written in C# and are compiled into a DLL called AccmanBackEndLibrary.DLL. They are executed by another program called the ConsoleSchedulerService.

The ConsoleSchedulerService uses Quartz under the covers, which is an open source scheduling API. You configure it with an XML file called quartz_jobs.xml. This file lists a series of ‘Quartz Jobs’, which correspond to each of the processes we want to run.

So, for example we have a quartz job called ActivateAssociate. This has an entry in the quartz_jobs.xml file. The entry lists various parameters that this process requires, as well as information about when the job should be executed.

We use ‘cron triggers’ to fire the processes at regular intervals, so for example, the ActivateAssociate process run every 30 seconds. Below is the entry from quartz_jobs.xml  for ActivateAssociate in its entirety.

Key Points

The main points of interest in the XML entry above are the following elements

  1. job-data-map: this is a list of name/value pairs used for setting any arbitrary parameters that our job might need. As an example, you can see that there is a parameter called LogDirectory with a value of C:\Logs\ActivateAssociate. Because of this, when the process is running, it writes its logs out to that directory. There are also other parameters for the database connections it needs (AccmanDatabase, AssociateDatabase), where transaction log files should be written to (TransactionLogs) and a few others. Other processes may have completely different parameters depending on their needs.
  2. trigger: this contains information about when the job should execute. As you can probably work out, the cron-expression entry that it contains gives information about when & how often the ConsoleSchedulerService should execute this job (Google “cron expressions” if you aren’t familiar with cron).
  3. A special mention also goes to the entry called ID. This is the transaction_type_id that all transactions in Accman uses.

So how does it all work?

All these new processes work essentially the same way as the perl processes. The steps are as follows:

  1. Some kind of user action (or even another automated process) writes a transaction record in the accman database. The transaction will have its tr_type_id column populated accordingly, and it will have a status of ‘waiting’ (as in waiting to be processed). Let’s say a HelpDesk user enables a student’s account using the AccmanWeb front end. Among other things, this will write a transaction record to the database with a tr_type_id value of 103 (ACTIVE_DIRECTORY_ENABLE_USER).
  2. The ConsoleSchedulerService is constantly triggering processes that are configured in quartz_jobs.xml, based on their cron schedules. At some point, it will trigger the ActiveDirectoryEnableUser process.
  3. That process then looks in the database for any transactions with a type of 103 that have a state of ‘waiting’.
  4. For any it finds, it processes the transaction (which in this case means connecting to AD, finding the user, and enabling the account).
  5. The processor writes any generic log messages to the directory configured in quartz_jobs.xml (under the LogDirectory setting).
  6. The processor writes any log messages that are specific to the transaction being processed into the directory specified by the TransactionLogDirectory setting in quartz_jobs.xml. This is the best place to look for any errors that might have occurred.
  7. When the transaction has been processed, it is marked as ‘processed’.
  8. If more than one transaction was found, they will all be processed.
  9. Once complete, the processor stops (until it is scheduled to run again according to the cron expression)

Starting / Stopping the ConsoleSchedulerService

The ConsoleSchedulerService is found in C:\ConsoleSchedulerService on accmanbe, and can be started by changing to that directory and running ConsoleSchedulerService.exe

At this point, it will have read quartz_jobs.xml and has started all the processes that it has been configured to run.

NB: the ConsoleSchedulerService also has its own log file, C:\Logs\SchedulerService.log which contains messages specific to this program.

To stop the ConsoleSchedulerService, simply type ‘exit’ at the command prompt above, and it will wait for any processes that are currently running to finish, and then exit gracefully.

Multiple Instances of ConsoleSchedulerService

Because ConsoleSchedulerService is a self-contained program, it’s entirely possible to run multiple instances – all with their own configuration – at the same time. This is useful if

  1. You want to run certain jobs on a one-off basis (rather than being on a schedule)
  2. A particular job runs better on another server.

The UpdateAllUsersADAttributesProcessor is a good example of why you might want to do this. This processor is configured to look at a certain location in ActiveDirectory, and for any accounts it finds there, reads the corresponding account details from the Accman database, recalculates what that user’s AD account values should be, and updates them. This is useful if we realize that certain AD attributes are incorrect. We can just move the accounts into a specified location in AD, configure the processor to look at that location, and let it run. It’s a powerful processor in that it has the ability to modify a lot of details (including changing the filestore and creating a mailbox) but because of this, we don’t really want it running all the time. It can also take a very long time to run (approx 1.5 hours to process 1000 accounts) on the old backend server. However on the new servers, it only takes a couple of minutes to run the same number of updates (because of the way the new servers are configured).

Therefore, we wanted this job to run on the new servers, and only when we decide to manually execute it.

In order to do this, we can take a copy of the ConsoleSchedulerService folder and place it anywhere we like. On the new Accman server (uos-p-ambe-01), there is a copy for this exact purpose, in C:\ConsoleSchedulerServiceAD.

All you need to do is ensure that the configuration file (ConsoleSchedulerService.exe.config) has sensible values (the only one that might need changing is the log file location). Then, ensure that the quartz_jobs.xml file has only the jobs in it that you want to run. In this case, we only have one, corresponding to the UpdateAllUsersADAttributesProcessor.

Now, whenever we want to run that one job individually, all we need to do is start that ConsoleSchedulerService.

Running an AD Update (or a Student Migration)

From time to time, we need to run a batch update of the AD Attributes. This could be for various reasons, for example Helpdesk notices that a group of 1000 students all have an incorrect email address. A more common occurrence is when we need to perform a student migration (usually once a day). This is usually done using the following method.

Use Case 1: A member of Operations identifies the accounts that need updating and moves them into an isolated OU (folder) in AD.

We isolate them like this so we can run the update on these users and these users only (without worrying about updating other unrelated accounts). We usually move them into an OU along the lines of OU=Import,OU=TempFix.

 

Or…

 

Use Case 2: We need to run a Student Migration

This is something that we do on a daily basis

  • Check that the file c:\accman\scripts\admigration\ad_student_migration.csv does not exist (if it does, rename it or delete it because it should not exist before we run the next step).
  • We run the C:\accman\scripts\admigration\ad_student_migration.pl script on accmanbe:

                cd c:\accman\scripts\admigration

                perl ad_student_migration.pl

  • This checks all student accounts in accman, and produces a CSV file called ad_student_migration.csv containing accounts that need to be migrated
  • Send the resulting file to Ops who will load the identified users into AD using some powershell scripts. They will usually be loaded into the OU=Import location in AD. Wherever they end up (Ops will tell you), set the ‘path’ parameter (see later) to this value.

Start the ConsoleSchedulerService

Let’s assume that the main ConsoleSchedulerService running on accmanbe is configured to process all transactions except type 110’s (this is currently the case).

Let’s also assume that we have a separate instance of ConsoleSchedulerService running on uos-p-ambe-01 for the sole purpose of processing 110’s i.e. its quartz-jobz.xml file only contains one entry, for transaction type 110 (again this is currently the case). This is located in C:\ConsoleSchedulerServiceAD and is currently NOT running.

Change to the directory C:\ConsoleSchedulerServiceAD and enter ConsoleSchedulerService.exe. This will start it running. You can check the logs (SchedulerService.log) to check it has started up without any errors.

Now, if the transaction we created earlier has a state of ‘waiting’ it should get processed straight away. If not, change the state to ‘waiting’ and the next time the UpdateAllUsersADAttributesProcessor is scheduled to fire, it will process the transaction.

When this happens, you will notice the following log files start getting written

C:\ConsoleSchedulerServiceAD\Logs\UpdateAllUsersADAttributesProcessor\ Salford.AccMan.BackEnd.Processes.UpdateAllUsersADAttributesProcessor.txt

This is a more general log file that the processor itself writes

And…

C:\ConsoleSchedulerServiceAD\Logs\TransactionLogs\1071077.txt

This contains very specific and detailed information relating to this particular transaction

Note: these transaction logs only get written to the disk once the transaction has been fully processed, so if the process takes a long time to run, don’t be too surprised if you don’t see the transaction log file appear for a while.

Once it has finished, you can check the logs to make sure everything looks OK, and if you check the transaction in the database, you’ll hopefully see it is now in the ‘processed’ state. If it’s in ‘pending’ and you are sure that processing has finished, there has probably been an error and it’s time to check the log files.

Move the accounts back

When you are happy that the accounts have been processed, remember they are still in a ‘temporary’ OU in AD, and they need to be moved back to their correct location. You can ask Ops to do this for you (Tom Greenwood). This is necessary because other AD-related processors require that the accounts are in correct location in order to work.

Query used by JanusDAO.GetJanusUpdates  Expand source 
SELECT distinct janus_users.EmpNumber, 
 UoS_Cards.SecurityTokenId,
 dbo.UoS_GetCardValue(UoS_Cards.SecurityTokenID, '32-bit Number') AS [32-bit Number],
 dbo.UoS_GetCardValue(UoS_Cards.SecurityTokenID, '56-bit Number') AS [56-bit Number],
 dbo.UoS_GetCardValue(UoS_Cards.SecurityTokenID, 'DESFire Number') AS [DESFire Number]
FROM 
 [Enterprise].[dbo].[UoS_Cards]
 INNER JOIN janus_user_SecurityToken
 ON UoS_Cards.SecurityTokenID = janus_user_SecurityToken.SecurityTokenId 
 INNER JOIN janus_users 
 ON janus_users.UserId = janus_user_SecurityToken.UserId
WHERE 
 (UoS_Cards.Status IS NULL) OR (UoS_Cards.Status = 0)

For each record it finds...

  1. Looks up the user in Accman (using the rollnumber which Janus knows)
  2. Looks up the user in AD and sets their ThirtyTwoBitNumber and FiftySixBitNumber attributes to the values retrieved from Janus
  3. If the above was a success, generate an UPDATE_TALIS_BARCODE transaction (type 126) passing the rollnumber and DESFireNumber as parameters.
  4. If the account is for a student:
    1. create an UPDATE_BANNER_BARCODE transaction (type 96) passing the rollnumber and DESFireNumber as parameters.
    2. create an UPDATE_LOXS_STUDENT_BARCODE transaction (type 107) passing the rollnumber and FiftySixBitNumber as parameters.
  5. Finally, removes the update request from the Janus database so that the next time the above SQL query is executed, it wont be processed twice. It uses the following SQL to do this:

DELETE FROM UoS_Cards WHERE SecurityTokenId = @SecurityTokenId

BULK_UPLOAD_PASSWORD_EMAIL

129.NET

At the end of the bulk upload process (AssociateBulkUploadController.cs) this single transaction is created with the name of the cohort management group and the name and email address of the cohort manager as parameters. 

 

The transaction is subsequently processed by C# process BulkUploadPasswordEmail.cs in order to retrieve the usernames and passwords of all the associates in the cohort and to generate a password protected Excel file containing these details.   The Excel file is then emailed to the cohort manager together with a second email which contains the password for the encrypted file.

 
ANONYMISE_BULK_UPLOAD_ASSOCIATE_DETAILS130.NET

A requirement for the AccMan Bulk Upload 2014 modifications is that the personal details of bulk upload associates held in the 'Associates.Associate' table are anonymised 90 days after the expiry of their 'EndDate'.  This transaction is a dummy transaction for the timer-driven C# process AnonymiseBulkUploadAssociateDetails.cs.

Each time the AnonymiseBulkUploadAssociateDetails.cs process anonymises a new associate account, it then creates a Type 131  transaction ANONYMISE_BULK_UPLOAD_ACCMAN_DETAILS which will subsequently trigger the anonymisation process in the AccMan (i.e. “mail” database) tables.
