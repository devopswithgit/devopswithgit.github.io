Introduction

Accman produces a vast amount of logging using a variety of methods. This page documents these and provides some suggestions as to how it can be improved.

Accman Back End

The Accman back end is a collection of processes (Quartz jobs) that either execute transactions found in the database, or simply run in a continuous loop. Quartz jobs tend to be fairly atomic i.e. there is one Quartz job responsible for one specific task such as creating an account in AD or resetting a user's password. All Quartz jobs execute on a schedule and the schedule of one Quartz job is independent of any other except there multiple jobs have to run in a certain order. For example, the SAP import consists of several Quartz jobs. The first job in the sequence runs at e.g. 09:00 and then the second in the sequence will be configured to run at 09:10 to give the first one time to complete. 

At a very high level a Quartz job is executed as follows:

  • ConsoleSchedulerService.exe - the main entry point for execution and when it starts, it loads the configuration which tells it which Quartz jobs are active and when to execute them. It creates an instance of ServiceProcess, which is assigned the task of looking for any waiting transactions for a specific type of job, and passes it the configuration details.
  • ServiceProcess - there are many instances of ServiceProcess, one for checking if there are any 'create AD account' transactions, one for checking for 'reset password' transactions and so on. It only checks for transactions at the time it is scheduled to do so. When it finds one or more transactions in a 'waiting' state, it executes the TransactionProcessor responsible for processing these types of transactions, passing it the ID of the waiting transaction. It does this in a loop, so if there are multiple waiting transactions, they are executed one after another. 
  • TransactionProcessor - there is one transaction processor for each type of transaction. For example, there is one called ActiveDirectoryNewUser and another called SetUserActiveDirectoryPassword. When executed by the ServiceProcess, it find the transaction based on the ID it was given and carries out the actual work involved.

All three levels of execution have their own log separate log files:

ComponentLog filePurposeNotes
ConsoleSchedulerServiceD:\AccmanLogs\SchedulerService.log

This is the highest level logging. If there is a serious error that isn't handled by the transaction processor, it usually finds its way into this log file as a 'last resort'. This file reaches its maximum file size very quickly so it's often difficult to find information (or important messages are quickly overwritten). It's generally recommended that only very important messages about the state of the applicacation (Accman) are logged here. When writing log messages, developers should bear in mind that the message is generally going to relate to a specific transaction being processed at the time so it is usually better to write it to the transaction log (see below).

This file uses Log4Net so this can be considered a 'proper' log file i.e. the maximum size is managed and new file is created when that size limit is reached.

If running under ConsoleSchedulerServiceForAlma or ConsoleSchedulerServiceForSapImport the filename is slightly different

ServiceProcessD:\AccmanLogs\<ServiceProcessName>\<ServiceProcessName>.log

There is one of these files for each type of transaction. They are stored in their own folder (see left) within D:\AccmanLogs. They are generally quite unhelpful because they only usually contain the following message:

Number of unprocessed ACTIVE_DIRECTORY_NEW_USER transactions: 3

They also print out their configuration parameters every time they run according to their schedule, resulting in a huge amount of repeated information.

They do not use Log4Net. Accman simply outputs the log message as if it was writing a basic text file, so the size is not managed. It is not uncommon to see these files grow to several gigabytes in size, with the same message repeated over and over again. They compress very well because of this, but because they contain so little useful information they usually just get deleted.

 
TransactionProcessorD:\AccmanLogs\TransactionLogs\<TransactionID>\<TransactionID>.txt

One of these files is created for each transaction that gets processed. There are literally millions of these files as a result and they are usually quite small because they only contain messages for that individual transaction e.g. "About to reset password for Joe Bloggs...". However, these are usually the most useful log files.

When a user of the Accman Front End requests a log file, it tries to find this file on the server, and then on a CIFS share (archive). If found, the contents is presented to the user which is often useful for problem solving.

We try to keep transaction log files for as long as possible but because they take up so much space we are limited to how long we can keep them. To reduce the disk space on the server, any transaction log file older than 30 days is archived onto a CIFS share. Then, after 6 months, it is deleted from the CIFS share.

Improvements

  1. Stop printing out configuration params every time a transaction is executed - greatly reduce the amount of unnecessary clutter in the files
  2. For messages such as Number of unprocessed ACTIVE_DIRECTORY_NEW_USER transactions: 0, don't bother printing it if the number is 0 and consider if this message is needed at all
  3. Consider compressing the files stored on the CIFS (but would require some way of retrieving zipped up log files - either automated or manual)
  4. Make logging use levels more appropriately e.g. DEBUG, INFO, ERROR. Quite often DEBUG is used for everything
  5. Make ServiceProcess and TransactionProcessor logs use Log4Net instead of being basic text files
  6. Consider if we even need ServiceProcess logs at all.
  7. Consider moving the TransactionProcessor log messages to the database instead of being file based
  8. ...
