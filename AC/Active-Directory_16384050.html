Introduction

Accman handles the following AD-related operations

  • Creates new staff/student/associate AD accounts 
  • New functional accounts 
  • Enabling account
  • Disabling account

Creating a New Account

Summary


ProcessTransaction Type IDScript / Class.NET / Perl
Triggered byRegister43register.plPerl
Executed byActiveDirectoryNewUser 93Salford.AccMan.BackEnd.Processes.ActiveDirectoryNewUser.NET


Process

  1. Calculate the field values (see the Excel spreadsheet below).
  2. Check that the password is of sufficient complexity and change it if necessary
    1. This is now redundant. It was a check made during the Transformation project when Accman wasn't generating passwords of sufficient complexity. The code is called (checkPassword()) but it always leaves the password intact because Accman generated acceptable passwords now.
  3. Find out where in the AD tree the user should be created. There are four different locations that could be used
    1. Functional: OU=Functional,OU=Users,OU=Non Datacentre,<ad-root>
    2. Staff: OU=Staff,OU=Users,OU=Non Datacentre,<ad-root>
    3. Student: OU=Students,OU=Users,OU=Non Datacentre,<ad-root>
    4. Unknown: OU=Other,OU=Users,OU=Non Datacentre,<ad-root> (if the account type cannot be determined)
  4. Set the password expiry policy via the PasswordNeverExpires property.
    1. Associate accounts: Password can expire (an account belongs to an associate if the username starts with 'AAA' or 'AAC')
    2. Other accounts: Password never expires
  5. Mark the account as enabled. 
  6. Non-Functional accounts only: 
    1. set the HomeDrive property to "F:"
    2. set the HomeDirectory property to the correct value:
      1. Students: <HomeFolderA>\\<username> or <HomeFolderB>\\<username> depending on whether their username ends in an even or odd number
      2. Staff: <StaffHomeFolder>\\<username>
  7. The new account is then created in AD 
    1. Note, the password is not set at this point
  8. For Staff and Student accounts, set the password using a secure connection
  9. Add the account to AD groups if there is a transaction parameter associated with the type-93 transaction and it has the name "Groups". Split the param value (it will be comma-separated) and for each group name, add the account to that group in AD. Note: groups are stored under 
  10. Student groups (Student accounts only):
    1. Student groups are found under OU=Provisioned,OU=Groups,OU=Non Datacentre,<ad-root>
    2. Remove the account from any previous programme-related AD groups (to cater for students switching programmes)
    3. Determine what the student's Program is using the AD Property called "Program" which will be something like "A/BCD/E"
    4. Find the corresponding AD group which will be something like "A-BCD-E". If the group doesn't exist, create it.
    5. Add the account to the group
  11. Set low-level attributes
    1. userAccountControl = 544
    2. pwdLastSet: this is a special integer value (essentially -1) whose value signals that the user will never have to change their password (see the list of Fields for more information).
  12. Security groups (note, these do not seem to be configured in production at time of writing so may not be necessary. This is used where we need to secure something to All staff and All students - Accman was supposed to populate this but I don't know if it does - in any case, a script runs on uos-p-admo-01 to fix up any accounts which are not a member of one of these groups)
    1. Staff: add to AD group specified by the quartz config param StaffSecurityGroup
    2. Students: add to AD group specified by the quartz config param StudentsSecurityGroup
  13. CreateExchange Mailbox
    1. Creates a type-115 transaction which creates the mailbox in Exchange.
    2. Associates: (where Mail.Person.SOURCE_CODE is "A"): only get a mailbox if their associate class permits it
    3. Staff: always get a mailbox
    4. The execute_at parameter is set to the current system time plus the number of minutes specified by quartz config param mailboxPause to allow AD replication to take place before the transaction fires
  14. Set Equitrac Initial Printer Credit 
    1. Note: applies to Students only, unless the quartz config param enableEquitracInitialPrinterCredit is set to "TRUE"
    2. Create a type-119 transaction, with the execute_at parameter set to the current system time plus the number of minutes specified by quartz config param equitracPause to allow AD replication to take place before the transaction fires.
  15. Create Home Folder
    1. For Staff and Students only
    2. If a transaction parameter called CreateHomeFolder exists and has the value "FALSE", do not proceed.
    3. Check if the folder already exists and only proceed if it doesn't
    4. Create the home folder with the path specified by the HomeDirectory property
    5. Create any sub folders specified using the HomeFolderSubDirs quartz config parameter. Currently set to Temp,PMail.
  16. Set permissions on Home Folder and its subdirectories
    1. Home Folder only
      1. First Set of rights
        1. FileSystemRights.FullControl
        2. InheritanceFlags.None
        3. PropagationFlags.NoPropagateInherit
        4. AccessControlType.Allow
      2. Second set of rights
        1. FileSystemRights.FullControl
        2. InheritanceFlags.ContainerInherit
        3. InheritanceFlags.ObjectInherit
        4. PropagationFlags.InheritOnly
        5. AccessControlType.Allow
    2. Subdirectories
      1. There doesn't seem to be any specific permissions set on home folder, even though the code can cater for differences. 
  17. Add to Novell AD group specified by quartz config param accountMigratedNovellGroup. - TG - Legacy - could be removed
    1. Creates a type-105 transaction to do this
  18. Add to the AD Group specified by quartz config param accountMigratedADGroup.
  19. Add to context-based group
    1. Find the account's default context (which is stored in Mail.DEPARTMENT_STATUS.DEFAULTCONTEXT) and looks something like this: pms.pm.bf.salford
    2. Note: the DEPARTMENT_STATUS table is joined to the ACCOUNT table using ACCOUNT.department_status_code = DEPARTMENT_STATUS.department_status_code
    3. Calculate the group name which is a concatenation of the following 
      1. The string "G-PE-"
      2. The third element of the default context e.g. "bf"
      3. another hyphen "-"
      4. The second element of the default context e.g. "pm"
      5. another hyphen followed by a dollar symbol "-$"
      6. First element of the default context e.g. "pms"
    4. The result is a group named something like "G-PE-bf-pm-$pms
    5. If the group does not exist, it is created first.
  20. Add to Password Sync group - TG - Legacy - could be removed
    1. Note: Applies to Students only
    2. The group name is configured as quartz config parameter passwordSyncGroup. Currently configured as ".PasswordSync.Systemadministration.Groups.Salford".
    3. Creates a type-105 transaction to do this
  21. Add to AD Script Enabled group - TG - Legacy - could be removed
    1. Note: Applies to Staff, Associate and Functional accounts (not Students)
    2. Name of the group is configured via quartz config param adScriptsEnabledGroup. Currently configured as ".Adscriptenabled.SystemAdministration.Groups.Salford".
    3. Creates a type-105 transaction to do this
  22. Add to Disable DLU group - TG - Legacy - could be removed
    1. Note: Applies to Staff, Associate and Functional accounts (not Students)
    2. Name of the group is configured via quartz config param disableDLUGroup. Currently configured as ".DisableDLU.SystemAdministration.Groups.Salford".
    3. Creates a type-105 transaction to do this

Note: Additional comments from Thomas Greenwood below

This isn't necessarily correct, AD Sync Script does not add the Zip Code to the users which Accman creates. AD Sync Script works on contacts. The user and contact are two separate entities (although they could be combined). Staff only have a user account. All Students have both a user account and a contact.

An example script which does alter attributes is the uid script which runs on uos-p-admo-01 every half hour. It looks for any staff and students who do not have uidNumber and gidNumber attributes. For each of these users it queries Mail.posixAccount for the users uidNumber and sets the corresponding AD attribute. It sets gidNumber to 1001 for all users. This is primarily used by Rick Odgen and may soon be used by the MAM - any POC will have to include this or students accounts created with the POC will not be able to access all CSE systems.

Uses of the above attributes in AD

There are several uses of the attributes in AD accounts they are at least:

Query based distribution Groups - used to contact cohorts of students (136 groups at time of writintg are listed in the spreadsheet here) - These probably actually use the contacts

 Dynamic Distribution Groups.xlsx

Global Address lists - forming the strcuture of the address book

Unix/Linux based systems for authentication - (CSE/MAM) (uidNumber and gidNumber specifically)

Equitrac - uosRfidCsnLong (Equitrac synchronises the number from AD regularly). This allows the user to log into the printer using their ID card.

USIR - USIR take a daily dump of all the staff and students in AD to allow them to collect email addresses, uosaffiliations and usernames. They use this information to provide a login and subscription service.

The user's new password. Must be at least 8 character long and must include at least:

  • one uppercase letter
  • one lowercase letter
  • one number
ExpireTRUE/FALSEWhether the user's password is set to expire. Default is FALSE but can be overridden by this parameter

Process

  1. Obtain a connection to Active Directory
  2. Look up the user's directory entry in AD, searching via IdentityType.SamAccountName (provided via the Username transaction parameter)
  3. If the user is not found, set the transaction's state to 'retry' and the ExecuteAt property to 5 minutes in the future (this is to allow AD replication to happen if necessary). The transaction will then fire again in 5 minutes. 
  4. If the user is found, call the SetPassword method on the user's directory entry object, and then the Save method.
  5. For security reasons, remove the NewPassword transaction parameter from the database and add an Expiry parameter to TRUE/FALSE to record any decisions that the code may have made (in the case that this parameter was not initially provided, and the default was used instead). 
    1. Note 1: Although the code does seem to try to do the above, I can't see where the changes are actually committed to the database so it might be that they aren't.
    2. Note 2: Also, note that the transaction param name is actually Expiry in step 5, not EXPIRE. So the transaction can end up having two different transaction params: EXPIRE and Expiry!
  6. Disconnect from AD.

Transaction Params

Param nameExampleDescription
UsernameAAA1234Username of the person whose password is being changed

Process

  1. Find the user's ADUserClassExtensions record in AD by their username.
    1. If the Username parameter has not been provided, first find their username in the Accman database using the AffectsID column in the TRANSACTION record.
  2. If the account cannot be found in AD, set the transaction's state to RETRY and the ExecuteAt value to 30 minutes in the future (to allow for replication delays)
  3. Set the Enabled property on the user's record to true and save the record.
  4. Create a type 97 transaction (UPDATE_JANUS_USER_DETAILS) to inform Janus that the account is enabled. The AffectsID property of the transaction is set to the user's AccID.
  5. Disconnect from AD.
