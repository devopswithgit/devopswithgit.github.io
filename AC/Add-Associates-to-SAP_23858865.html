Details

Associates are not currently fed into SAP by Accman because they are not generally paid by the university. However some associates are in SAP for various reasons. They are in SAP because they have been entered manually. Examples of these are employees of a company called SPD Ltd. They have been manually entered into SAP because they are allowed to claim travel expenses from the university through the Agresso system, and they can't do this unless they're in SAP too.

Matthew Blundell has recently requested that Accman includes these associates in the feed to SAP (via the SAP Export Email process) to reduce the amount of work they have to do. 

After some initial investigation, this might be harder to accomplish than originally thought. The purpose of writing this down here is to start a discussion regarding some functionality that may (or may not) make it into Accman.

Why Aren't Associates Included in the SAP Export?

When Accman selects accounts to send to SAP it uses an SQL query that simply doesn't pick up any associate accounts. Associates don't have personnel numbers for a start, and the query has an inner join that required a personnel number to be in the ROLLMAP table. They also have a different status code. This is one of S (Staff), T (Temporary), A (Student) or L (Student). SPD Ltd's staff have a status code of T.

There are other differences but the main point is that associate data is very different to staff data.

A Specific Example

Sabrina Testoni is an Associate working for SPD Ltd. She was created in Accman as an associate, and has the details shown below. Note, this account is in no way linked to SAP. 

In order to get her details into SAP, HR created a new account for her (in SAP, not Accman). This gave her a personnel number of 02034319. Shortly after, the SAP Import ran. SAP Import could not find any existing PERSON records linked to the personnel number, so created a completely new person record. Note, SAP Import does not create Account records. It just creates Person records.

There is no way for the automated SAP Export process to pass Sabrina's username and email address to SAP. To do this, it needs to find an Account record linked to a Person record, linked to a Rollmap record which contains the personnel number. As you can see in the diagram, we could conceivably achieve this by linking the Account on the left with the Person on the right of the red line, and changing some other values such as the account type.

If we are to include a select group of Associate accounts in the SAP export, this is a process change that needs to be discussed further. 

Options

Feel free to add to this list.

  1. Heavily modify the SAP Export process to somehow identify SPD associates using some arbitrary rules (and also link the account and person records)
  2. Discard the Associate's Person record (left hand side) and point the Account at the staff's Person record (right hand side).
  3. Add a new associate class called e.g. Paid Associates which we can use to identify associate accounts that need pushing into SAP (will still need some magic to link the two sides of the above diagram together)
  4. Add functionality to Accman to somehow convert an associate account to something resembling a staff account. A user would need to specify the personnel number at the time of doing this. This would prevent users having to manually add accounts in SAP, which is what the users have been doing up until now. 
  5. Investigate the Reconcile process to see if that could be modified in some way to link the two sides.
  6. Any other suggestions
