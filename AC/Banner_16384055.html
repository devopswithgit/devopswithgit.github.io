Account mapping between Banner and Accman 

Introduction

This section describes interactions between baseline Banner and the bespoke SYRACCS table, which maintains the mapping between Banner and Accman accounts.

Database tables

 The SYRACCS table contains the following fields:

Field NameDescriptionPermissible   Values



 SYRACCS_ROWNUMInternal record identifierNumeric,  up to 99999999
SYRACCS_USERNAMEPre-allocated account identifierFrom Accman USERNAME field on ACCOUNT table; note that this is limited to 6 characters
SYRACCS_ACCMAN_STATUS
A / N / R ; default   is N
SYRACCS_PIDMInternal Banner identifierFrom Banner SPRIDEN_PIDM field on SPRIDEN table
SYRACCS_SIS_STATUS
A / N ; default is N
SYRACCS_INIT_PASSWORDInitial passwordGenerated character string, or NULL
SYRACCS_DEPT_CODEBanner department (a.k.a. school) codeAny contained in Banner STVDEPT table
SYRACCS_LEVL_CODEBanner level-of-study codeUG or PG
SYRACCS_ACTIVITY_DATEDate and time record was last modifiedAny valid date and time

 How records are created in SYRACCS

The Perl scripts allocateNew.pl and allocateLib.pl generate records in SYRACCS; Accman accounts to create or update are identified by allocateNew.pl and the subroutine newAccountToSYRACCS in allocateLib.pl creates or updates SYRACCS records . A common approach is used to create both undergraduate and postgraduate accounts, although each is handled separately.

The Accman account table acts as the data source and undergraduate / postgraduate accounts are identified thus:

UG : status_code < 'P' AND department_status_code != 'XXD' AND state = 'new' AND username > 'AAA99999'

PG : status_code = 'P' AND department_status_code != 'UIP' AND state = 'new' AND username > 'AAA99999'

The key point to note is that SYRACCS entries are created by Accman.

How records are removed from SYRACCS

The Perl script deletesyraccs.pl removes records from SYRACCS that are no longer required. The latter are identified by a state value of 'deleting' in the Accman account table.

For each username found in account where state is 'deleting' and status_code is less than or equal to 'P', all SYRACCS records where SYRACCS_USERNAME equals username are deleted.

How SYRACCS is updated by Banner

When a new SYRACCS record is created in Banner, it is initially not attached to a student. There are several processes that will cause a student to be associated with a SYRACCS record and this section details what they are.

Notification of applicant response

Whenever an applicant's final response is received, a database trigger confirms that the response status is 'A' (firm acceptance); this value is stored in the database field SKRUCCR_SSDT_CODE_AP_RESP. The trigger - T_SAL_ALLOCATE_UNAME_SKRUCCR - then calls the Oracle database stored procedure P_SAL_ALLOCATE_UNAME_SKRUCCR, passing in their applicant number and the unique identifier for the study choice the response is against.

The stored procedure identifies the following information about the applicant:

DescriptionSource


Internal Banner identifierSARADAP_PIDM
Level of studySARADAP_LEVL_CODE
Department codeSARADAP_DEPT_CODE

This information is then passed to the Oracle database stored procedure P_SAL_ALLOCATE_USERNAME1, which is documented below.

Change of student registration status

When a student's registration status is changed, a database trigger confirms that the new registration status is either 'RE', 'RS' or 'RP'; this value is stored in the database field SFBETRM_ESTS_CODE. The trigger - T_SAL_ALLOCATE_UNAME_SFBETRM - then calls the Oracle database stored procedure P_SAL_ALLOCATE_UNAME_SFBETRM, passing in their internal Banner identifier (PIDM) and the term code associated with the updated registration record.

The stored procedure identifies the following information about the student:

DescriptionSource


Internal Banner identifierSGBSTDN_PIDM
Level of studySGBSTDN_LEVL_CODE
Department codeSGBSTDN_DEPT_CODE

This information is then passed to the Oracle database stored procedure P_SAL_ALLOCATE_USERNAME1, which is documented below.

Allocation of Accman username to Banner applicant / student - P_SAL_ALLOCATE_USERNAME1

The Oracle database stored procedure P_SAL_ALLOCATE_USERNAME1 performs the necessary processing to associate a student or applicant (identified by PIDM), studying at a specified level in a particular department, with an Accman account. The following flowchart illustrates the processing logic.

Batch allocation of Accman username to Banner applicant / student - P_SAL_ALLOCATE_USERNAME_BATCH

The Oracle database stored procedure P_SAL_ALLOCATE_USERNAME_BATCH acts as a 'sweep up' process for any students that have failed to have a SYRACCS account attached by P_SAL_ALLOCATE_USERNAME1. This is typically caused by no unallocated SYRACCS record being available with the appropriate department code and level code. New unallocated accounts will be created by allocateNew.pl, as described above.

The students that are to be allocated accounts via the batch process are identified by the following criteria which comprise a SQL driving cursor in P_SAL_ALLOCATE_USERNAME_BATCH; note that the procedure takes a Banner term code as an input parameter.

Any student with a status of 'AS' (in SGBSTDN_STST_CODE) in the term identified by the input parameter P_TERM, or later, that has a corresponding registration record (SFBETRM), based on PIDM (SFBETRM_PIDM) and term code (SFBETRM_TERM_CODE), with a registration status (SFBETRM_ESTS_CODE) that is not in the GTVSDAX internal code group 'NETWORK_ACCESS' with internal code 'NOT_HERE' and that does not already have a SYRACCS record assigned to the same PIDM (SGBSTDN_PIDM), department code (SGBSTDN_DEPT_CODE) and level (SGBSTDN_LEVL_CODE).

Each combination of PIDM, department code and level code identified above is passed to P_SAL_ALLOCATE_USERNAME1, which performs the actual account allocation.
