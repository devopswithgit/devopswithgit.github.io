Accman : Unit Testing

Created by Unknown User (aaa45816), last modified on Jun 14, 2016

Unit tests should be written to represent a business logic function.

Once a unit test is written and passed, it should not change. The implication of changes to a unit test may indicate a feature change. However SOLID principals should ideally be adhered to and so if a feature change is required, it is more likely that business requirements for that feature were not clear enough in the first instance.

The SOLID 'O' princpal suggests logic should be extended not modified. If the logic needs to be changed so that it does something completely different to the original specification then this can be deemed a new feature and the old feature retired and no longer tested.

Benefits of unit testing AccMan

  • Code is written in testable and re-usable components
  • If an issue is raised by the user or via exception logging etc, the developer can write a unit test expecting the reported exception and diagnose it easily
  • Business requirements are verified and potentials omissions or flaws in design are highlighted at the first stage of development
  • Testers can be provided with a list of unit tests and outcomes from Dev environment to compare with QA and UAT testing to aid and to provide confidence
  • Not all changes to features may impact the user directly. A bug fix may only be required in a back end process. In this case, a valid unit test can prove that the bug is fixed and it's release will not impact on such critical components
  • Critical AccMan processes can be built and deployed with confidence of passed tests.
  • Code reviews have a standard structure starting with the feature itself, the validity of the test's assertions and it's outcome to business requirements.
  • Team members can navigate unit tests easily and become proficient and familiar with the code base, resulting in a stronger Skills Matrix
  • Duplicate code and features are avoided as test names should be easily recognisable as existing features
  • Tests can be ignored as part of the build if the development of it is not complete, allowing sustained confidence level between releases

AccMan Unit Tests review

Currently there are select unit test projects in the AccMan solution. They are well structured and perform a good set of positive tests. There is a large percentage of code blocks not tested which should be to prove a feature works.

What's good

  • The tests are fairly well structured and concise without too many multiple asserts per test.
  • Good level of Active Directory interfacing and processing logic is tested well
  • Good level of MVC controllers tested

What could be better and why

  • The naming of tests could be improved to reflect the actual business requirement and the feature which satisfies it
  • The class design of the AccManLibrary and AccManBackendLibrary initialise and manage it's own instance of Common.Logging Log manager. This is difficult to test.
    The improvement here would be to have a process or service class design which would have it's "log provider" implementation injected to it. This would allow greater testability and easier log management.
  • Controller actions should be tested with more specific requirements in mind, e.g. AssociateRequestControllerTest.cs has a test named "Complete", which only proves that a ViewResult is returned. This is a generic test and the completion of the action should be tested instead.   • More features of the AccMan Library need to be unit tested. See the code coverage levels below e.g. accmanlibrary.dll is only ~7% covered
