Introduction

The Accman Codebase includes three main 'solutions'

They have historically been located on the older subversion repository. For various reasons, these have been moved to the newer subversion repository and renamed. The old and new locations are shown in the following table.

Decomissioning

We will not be deleting the old source code because it contains a lot of history which we don't want to lose. It also contains lots of extra branches that in theory are no longer needed but since we are only moving the current production code (one branch per project).

However, in order to prevent anyone from accidentally trying to work on the old codebase, they have all been renamed (so commits will fail). The root of each project has been prepended with the word: DECOMISSIONED_ e.g. svn://isd-sysarch01.isdads.salford.ac.uk/UoS_IdM/DECOMISSIONED_Accman.NET/branches/Associates

Other Branches

There were several other branches in the old location that contained changes which had not yet been released to production. However they were inappropriately named and some changes seemed to be together in the same branch making it difficult to release them separately, or to work out what changes were in which branches at all. These branches will be recreated from scratch by manually creating individual branches in the new repository and then making the necessary code changes. Since the changes are not major this will not be too difficult and should make it easier to manage the release of these changes to production when they are ready. They are

1)    Associate Cohort Management Changes (Slow Performance and “re-enable AccMan and AD accounts following change of start date”)

Modification in response to ITBM Case 242283. It was taking around half an hour for the "Manage Associate Cohorts" form to open and to return from it in AccMan Front End.

To Test
1. The time taken between selecting the "Manage Associate Cohorts" and the page appearing was around a quarter of an hour. It should now be almost immediate.
2. A general regression test of the Manage Associate Cohorts functionality in AccManFE to ensure that the modification has not broken anything,
•   
Modification in response to ITBM Case 251440.  Modification to re-enable AccMan and AD accounts following a change of start date for a cohort.

To Test
1. In AccManFE, change the start date for a pre-existing cohort and save the change. Expected Result: A transaction type 83 should exist for each of the associates in the cohort.
2. Start the ConsoleSchedulerService. Expected Result: a transaction type 103 should be created with the ExecuteAt date set to the new Start Date for each of the associates in the cohort.
3. Start the EnableDisable.pl perl script. Expected Result: a transaction type 13 should be created with the ExecuteAt date set to the new Start Date for each of the associates in the cohort.

See http://uos-d-dbld-01:8090/display/AC/CohortRescue+SVN+Branch for further details.



2) Posix Changes for AD

This RFC1567 is about removing the LDAP servers (i.e. Novell) and making the appropriate changes to AccMan in response.
In this particular case, there is an issue to do with the POSIX account UIDNumbers that AccMan pushes downstream for the computer science students who use a ubuntu dual-boot set-up and require the UIDNumber to be present as an attribute in order for this to work.
The required changes are:
•    add posix account id to Account object
•    update account DAO to retrieve posix account id
•    amend create new AD account back end process to add posix account id value into AD
The code change writes the UIDNumber generated as part of the resgister.pl script to AD during AD account creation.

To test:
Check that the UIDNumber attribute has been added to a newly created AD account, and that the value matches that allocated to the account in the posixaccount database table in AccMan.

See http://uos-d-dbld-01:8090/display/AC/RFC1567%3A+LDAP+Removal for further details

4) LoXS changes

The initial specifications for the LoXS interface turned out to be incorrect. As such, it transpires that while additions can be made to the system, changes and deletions cannot.  As such, there is a need to change the LoXS interface to allow for these changes.

In real terms, this means altering the LoXS DAO object such that:
•    if no card number is supplied, it is assumed that the action is disable
•    in this case, the card number is retrieved from the LoXS database (SELECT Kaartid FROM Gebruiker WHERE Sleutelveld = @rollnumber)
•    the card status is set to 0 for an active card and 1 for an inactive card (note that this is logically inconsistent)
To test, create a type 107 transaction as follows:
•    New User: ROLLNUMBER parameter = rollnumber and CARDNUMBER parameter = 56-bit card number
•    Update User: ROLLNUMBER parameter = rollnumber and CARDNUMBER parameter = new 56-bit card number
•    Disable User: ROLLNUMBER parameter = rollnumber and no/blank CARDNUMBER parameter

Note that, as there is no LoXS test system, testing has been successfully performed against the live system using a single known user (rollnumber 304476, 56-bit number 00036070408583472132) who was disabled in Janus.

See http://uos-d-dbld-01:8090/display/AC/SWR0467+LoXS+Interface for further details.

5) 233462: Staff changes on the SAP system to not propagate departmental changes to the AD system.
This fix requires minor changes to sap_import.pl to create a flag that gets set if the department code changes, and then create a type 121 transaction (which updates the AD department field) if that flag is set.
To test, change the department code in the sap_import table, run the sap_import.pl script, and see if a type 121 transaction is generated.  The type 121 transaction is already live and working, so no testing of this element is required.
                See http://uos-d-dbld-01:8090/display/AC/204752+Staff+AD+Department+Changes for further details.

Backout process for all of the above is to revert the DLL files/perl scripts as per usual.
